# 01 - Claude Code 项目总览与架构导图

## 1. 项目定位与技术画像

### Claude Code 是什么

Claude Code 是 Anthropic 官方推出的 AI 编程助手 CLI 工具。它以终端为界面，通过 Agent Loop（LLM 调用 → 工具执行 → 结果回传 → 循环）模式，让 Claude 模型能够自主读取文件、编辑代码、执行命令、搜索代码库，实现全自动化编程协作。

### 核心技术栈

| 技术 | 用途 |
|------|------|
| **TypeScript** | 主语言，~512,664 行代码 |
| **React + Ink** | 终端 UI 渲染（自定制 Ink 引擎） |
| **Bun** | 原始编译器（提供 `feature()`、`MACRO` 等编译时内建函数） |
| **esbuild** | 从源码重建的备选打包器 |
| **Anthropic SDK** | 与 Claude API 的流式交互 |
| **Zod** | Schema 验证 |
| **lodash-es** | 工具函数 |

### 代码规模

```
~1,884 个 TypeScript/TSX 源文件
~512,664 行代码
40+ 内置工具
80+ 斜杠命令
108 个 Feature-gated 模块（Dead Code Eliminated）
运行时要求：Node.js >= 18
```

版本：`2.1.88`（`package.json:3`）

## 2. 编译与构建流程

### 2.1 构建脚本概览

`package.json` 定义了 4 个脚本（`package.json:8-11`）：

```
prepare-src  → 预处理源码（替换 Bun 特定语法）
build        → prepare-src + esbuild 打包
check        → prepare-src + tsc 类型检查
start        → node dist/cli.js
```

### 2.2 核心编译挑战：Bun 编译时内建函数

Claude Code 原本使用 **Bun** 运行时编译。Bun 提供了两个编译时内建函数：

1. **`feature('FLAG')`** — 从 `bun:bundle` 导入（`stubs/bun-bundle.ts:2`），在 Bun 编译时根据 Feature Flag 返回 `true/false`。返回 `false` 的分支会被 **Dead Code Elimination（DCE）** 完全移除。
2. **`MACRO.VERSION`** 等 — 编译时字符串常量注入。

这就是为什么 npm 发布的 `cli.js` 约 12MB，而源码有 512K 行——108 个 Feature-gated 模块被 DCE 移除了。

### 2.3 prepare-src.mjs 的转换策略

`scripts/prepare-src.mjs`（115行）执行三项核心转换：

```
源码文件
  ├── import { feature } from 'bun:bundle' → import { feature } from '../stubs/bun-bundle.js'
  │     （stub 中 feature() 永远返回 false）
  ├── MACRO.VERSION → '2.1.88'  （字符串字面量替换）
  └── MACRO.BUILD_TIME → ISO时间戳
```

### 2.4 build.mjs 的多轮迭代构建

`scripts/build.mjs`（245行）采用独特的**迭代式构建**策略：

```
Phase 1: 复制 src/ → build-src/（工作副本）
Phase 2: 批量替换 feature('X') → false, MACRO.X → 字面量
Phase 3: 创建入口包装器
Phase 4: 最多 5 轮迭代：
  ├── esbuild 打包尝试
  ├── 收集 "Could not resolve" 错误
  ├── 为缺失模块自动生成 stub
  └── 重试
```

这种设计说明：即使 `feature()` 返回 `false`，TypeScript 的 `import` 语句仍然存在，需要 stub 文件来满足模块解析。

### 2.5 tsconfig.json 配置要点

`tsconfig.json` 关键配置：

```json
{
  "target": "ES2022",        // 使用顶层 await、class fields 等现代语法
  "module": "ESNext",        // ESM 模块系统
  "moduleResolution": "bundler", // 打包器模式解析
  "strict": false,           // 非严格模式（大项目务实选择）
  "jsx": "react-jsx",        // React JSX 转换
  "paths": {
    "bun:bundle": ["stubs/bun-bundle.ts"],  // Bun 内建模块别名
    "src/*": ["src/*"]        // 路径别名
  }
}
```

### 2.6 stubs/ 目录：Bun 运行时替代

```
stubs/
├── bun-bundle.ts    # feature() → false（Feature Flag DCE）
├── global.d.ts      # MACRO 全局类型声明
├── macros.d.ts      # 宏类型声明
└── macros.ts        # MACRO.VERSION 运行时替代
```

`stubs/bun-bundle.ts` 只有 4 行，却决定了 108 个模块的生死：

```typescript
// stubs/bun-bundle.ts
export function feature(_flag: string): boolean {
  return false  // 所有 Feature Flag 全部关闭
}
```

### 启动优化技巧

`cli.tsx` 采用了**动态导入 + 快速路径**策略（`cli.tsx:33-298`）：

```typescript
// --version 快速路径：零模块加载
if (args[0] === '--version') {
  console.log(`${MACRO.VERSION} (Claude Code)`);
  return;  // 直接退出，不加载任何模块
}

// 其他路径：按需动态导入
const { main: cliMain } = await import('../main.js');
```

这种设计确保了 `claude --version` 命令在 < 50ms 内完成，而完整启动（加载 React、Ink、所有工具）需要 1-3 秒。

## 7. 关键设计决策

### 7.1 为什么选择 React/Ink 做终端 UI

传统 CLI 工具使用 `inquirer` 或原生 `readline`，但 Claude Code 需要实时渲染：
- 工具进度实时更新（旋转 spinner、进度条）
- 流式文本输出（逐字显示 LLM 响应）
- 并行工具结果同时展示
- 复杂布局（权限对话框、Diff 视图、虚拟滚动）

React + Ink 提供了声明式 UI、组件复用、状态驱动渲染，这些是命令式终端渲染无法优雅实现的。

Claude Code 甚至**自定制了整个 Ink 引擎**（`src/ink/`, 48个文件），添加了焦点管理、文本选择、虚拟滚动、鼠标点击等高级功能。

### 7.2 为什么使用 AsyncGenerator 做流式处理

`query()` 返回 `AsyncGenerator` 而非 `Promise`，原因：
- **端到端流式**：从 API SSE 事件到 CLI 输出，零缓冲
- **消费者自主**：调用者可以逐事件处理、中断、或忽略
- **组合性**：`for await...of` 天然支持管道式处理
- **错误传播**：`throw` 自然沿 generator 链向上传播

### 7.3 为什么使用 Feature Flags + DCE

Bun 的 `feature()` 是编译时函数，返回 `false` 的代码会被完全移除：
- **包体积**：12MB 而非 50+ MB
- **启动速度**：108 个模块不参与模块求值
- **安全隔离**：内部功能（ant-only）无法从外部访问
- **渐进发布**：新功能通过 GrowthBook Feature Flag 控制

代价是：外部构建需要 stub 所有 feature-gated 模块。

### 7.4 为什么自建状态管理而非使用 Redux

`src/state/store.ts` 实现了一个极简的 Observable Store（仅 ~50 行）：

```typescript
type Store<T> = {
  getState: () => T
  setState: (updater: (prev: T) => T) => void
  subscribe: (listener: Listener) => () => void
}
```

配合 `useSyncExternalStore` 实现 React 集成。这比引入 Redux/Zustand 更轻量，且：
- 无需中间件（Agent 状态更新是同步的）
- 无需序列化（终端 UI 不需要时间旅行调试）
- 更好的 TypeScript 类型推导
