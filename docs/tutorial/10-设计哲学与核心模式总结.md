## 2. 十二大核心设计模式

### 模式 1: AsyncGenerator 流式管道

**核心文件**: `src/query.ts`

```
query()                    → Agent Loop 主循环
QueryEngine.submitMessage() → 会话生命周期
StreamingToolExecutor      → 工具并发执行
withRetry()                → API 重试
```

**设计哲学**: 从 API 的 SSE 事件到工具的进度消息，再到 SDK 的流式输出，整个管道是零缓冲的。`AsyncGenerator` 实现了生产者-消费者的优雅解耦——生产端可以 yield 任何粒度的事件，消费端可以用 `for await...of` 逐事件处理。

**为什么比 Promise 好**:
- 流式输出：用户看到的是逐字显示，而非等待完成后一次性输出
- 中断性：调用者可以随时 `.return()` 终止生成器
- 组合性：`yield*` 语法让生成器可以管道式嵌套

### 模式 2: Feature Flags + Dead Code Elimination

**核心文件**: `stubs/bun-bundle.ts`, `src/tools.ts`

```
feature('FLAG') → false  (编译时)
    │
    └── 所有被 feature() 门控的代码块被 DCE 完全移除
        ├── 108 个模块从外部构建中消失
        ├── 包体积: 50+MB → 12MB
        └── 启动时间: 减少 60%+
```

**设计哲学**: 这不是简单的配置开关——这是编译时多态。`feature()` 在 Bun 编译时被替换为 `true/false`，然后 tree-shaking 移除死代码。这意味着：
- 外部用户无法访问内部功能（不仅仅是"关闭"，而是"不存在"）
- 安全隔离在二进制层面保证（反编译也无法恢复被删除的代码）
- 包体积和启动性能是编译时优化，不是运行时开销

**代价**: 外部构建需要 stub 所有 feature-gated 模块，导致构建系统复杂化（`scripts/` 的 4 个构建脚本）。

### 模式 3: buildTool() 工厂 + Fail-Closed 默认

**核心文件**: `src/Tool.ts:757-792`

```typescript
const TOOL_DEFAULTS = {
  isEnabled: () => true,
  isConcurrencySafe: () => false,     // 默认不安全
  isReadOnly: () => false,             // 默认非只读
  isDestructive: () => false,
  checkPermissions: () => Promise.resolve({ behavior: 'allow' }),
}
```

**设计哲学**: "保守的默认值"。每个新工具必须**显式声明**它安全——不声明则默认为不安全。这种 fail-closed 设计确保了：
- 新工具不会意外获得并发安全性
- 权限检查默认放行（由通用权限系统兜底）
- 安全相关属性必须显式覆盖

### 模式 4: Observable Store + Selector 订阅

**核心文件**: `src/state/store.ts` (34行), `src/state/AppState.tsx`

```typescript
// 极简 Store（34 行代码）
createStore(initialState, onChange) → { getState, setState, subscribe }

// React 集成
useAppState(selector) → useSyncExternalStore(store.subscribe, get, get)
useSetAppState() → store.setState  // 永远不触发重渲染
```

**设计哲学**: "做减法"。不用 Redux（无中间件需求）、不用 Zustand（无 devtools 需求）、不用 Immer（纯 spread）。34 行代码实现了：
- 不可变更新
- 引用相等性优化
- 选择器级细粒度订阅
- onChange 副作用回调

### 模式 5: 分区并发（Partition Concurrency）

**核心文件**: `src/services/tools/toolOrchestration.ts`, `src/services/tools/StreamingToolExecutor.ts`

```
partitionToolCalls(toolCalls)
    │
    ├── 连续安全工具 → 合并为一个并行批次
    │     [Read(a), Read(b), Read(c)] → Promise.all([call(a), call(b), call(c)])
    │
    └── 不安全工具 → 独占串行批次
          [Bash(rm)] → await call(rm)

StreamingToolExecutor:
  ├── 无工具执行中 → 任何工具可启动
  ├── 有安全工具执行中 → 只允许安全工具加入
  └── 有独占工具执行中 → 不允许任何新工具启动
```

**设计哲学**: **实例级别的并发安全**，不是类型级别。`isConcurrencySafe(input)` 接受输入参数——同一个 BashTool，`git status` 是安全的，`rm -rf /` 不是。这种粒度比 Java 的 `synchronized` 方法注解更精确。

### 模式 6: 三层上下文压缩

**核心文件**: `src/services/compact/`

```
Layer 1: MicroCompact   → 每轮自动执行（字符串替换，零成本）
Layer 2: AutoCompact    → token > 阈值时触发（LLM 摘要，中等成本）
Layer 3: ReactiveCompact → API 返回 prompt-too-long 时触发（紧急恢复）
```

**设计哲学**: "预防优于治疗"。压缩时机选择在 API 调用**之前**（`src/query.ts:454`），避免了：
- 先触发 API 错误（400 prompt-too-long）再恢复的昂贵往返
- 用户看到错误消息的体验中断
- 重试导致的额外 token 消耗

### 模式 7: Command Pattern + 动态分发

**核心文件**: `src/commands.ts`, `src/commands/`

80+ 斜杠命令通过注册表模式管理：
- 每个命令导出 `Command` 对象（name, aliases, handler, description）
- `getCommands()` 收集内置命令 + 插件命令
- `processUserInput()` 分发到对应 handler

**设计哲学**: 命令的增删不影响分发逻辑——新命令只需在 `commands/` 目录下添加文件并注册。

### 模式 8: Hook 生命周期

**核心文件**: `src/services/tools/toolHooks.ts`, `src/schemas/hooks.ts`

```
PreToolUse  → 工具执行前（可修改输入或阻止）
PostToolUse → 工具执行后（可处理结果）
SessionStart → 会话开始
SessionEnd → 会话结束
PreCompact → 压缩前
PostCompact → 压缩后
Stop → Agent Loop 轮次结束
```

**设计哲学**: "事件驱动扩展"。Hook 系统让外部代码（插件、组织策略）可以在不修改核心逻辑的情况下扩展行为。四种 Hook 类型（command、prompt、http、agent）覆盖了从 Shell 脚本到 LLM 评估的全部需求。

### 模式 9: Bootstrap Singleton + React State 分层

**核心文件**: `src/bootstrap/state.ts`, `src/state/AppStateStore.ts`

```
Bootstrap State（单例，命令式）
  ├── 成本追踪、Token 计数、Session ID
  ├── 不触发 React 重渲染
  └── 适合在 Agent Loop 内更新

AppState（Observable Store，响应式）
  ├── 权限上下文、任务状态、MCP 连接
  ├── 触发 React 重渲染
  └── 适合驱动 UI 组件
```

**设计哲学**: "关注点分离"。核心业务数据不需要 React，UI 状态不需要在 Agent Loop 内。这种分离避免了：
- React 重渲染在热路径上的性能开销
- 循环依赖（Agent Loop 不依赖 React）
- 测试复杂度（核心逻辑可以不启动 React 测试）

### 模式 10: Branded Types + DeepImmutable

**核心文件**: `src/types/ids.ts`, `src/types/utils.js`

```typescript
type SessionId = string & { __brand: 'SessionId' }
type AgentId = string & { __brand: 'AgentId' }

type DeepImmutable<T> = {
  readonly [K in keyof T]: DeepImmutable<T[K]>
}
```

**设计哲学**: "让编译器捕获错误"。`SessionId` 和 `AgentId` 在运行时都是 `string`，但在编译时不能互换——防止了 `getSessionId()` 的返回值被错误传给需要 `AgentId` 的函数。`DeepImmutable` 确保了 `AppState` 的所有嵌套属性在类型层面不可变。

### 模式 11: Lazy Schema（延迟求值）

**核心文件**: `src/utils/lazySchema.ts`

Zod Schema 在工具注册时不立即求值——只在被 API 调用时才计算。这避免了 40+ 工具注册时的批量 Schema 构建开销，将成本延迟到实际使用时。

### 模式 12: Fire-and-Forget 后台任务

**核心文件**: `src/query/stopHooks.ts`

```
Stop Hook 执行后:
  ├── executeAutoDream()         → 后台思考（不阻塞）
  ├── extractMemories()          → 记忆提取（不阻塞）
  ├── executePromptSuggestion()  → 建议生成（不阻塞）
  └── 所有任务 fire-and-forget
```

**设计哲学**: "不阻塞主路径"。后台优化任务（思考、记忆、建议）不应影响用户的主交互体验。它们静默执行，结果在下次轮次时注入。

## 4. 架构权衡

### 4.1 选择了什么

| 选择 | 理由 |
|------|------|
| **React/Ink 终端 UI** | 声明式 UI、组件复用、实时流式渲染 |
| **AsyncGenerator** | 端到端流式、零缓冲、可中断 |
| **Feature Flags + DCE** | 安全隔离、包体积优化、编译时多态 |
| **自建状态管理** | 零依赖、34行代码、满足所有需求 |
| **分区并发** | 安全且高效的工具并行 |
| **三层压缩** | 预防优于治疗、梯度防线 |

### 4.2 放弃了什么

| 放弃 | 理由 |
|------|------|
| **Strict TypeScript** | 512K 行代码严格模式的迁移成本过高 |
| **Redux/Zustand** | 无中间件需求、无 devtools 需求 |
| **标准 Ink** | 不支持鼠标、选择、虚拟滚动等高级功能 |
| **简单的 CLI 库** | 无法满足实时流式、并行工具、虚拟滚动等需求 |
| **用户完全自主** | 远程控制、Undercover 模式、不可退出遥测 |

## 5. 学习路线图

### 5.1 理解 Agent 系统的核心

要理解 Claude Code（或任何 AI Agent 框架），你需要掌握：

```
1. Agent Loop: while(true) { think → act → observe → repeat }
2. Tool System: 安全的能力边界 + 并发调度
3. Context Management: 上下文窗口有限 → 必须压缩
4. Streaming: 端到端流式 → 用户体验
5. State: 不可变 + 响应式 → UI 一致性
```

## 6. 结语

Claude Code v2.1.88 是一个 512K 行的 TypeScript 项目，代表了当前生产级 AI Agent 框架的最高工程水平。它的核心创新不是某个单一算法，而是一系列务实的架构决策的累积效果：

- **AsyncGenerator 贯穿始终**的流式管道
- **Feature Flags + DCE** 的编译时安全隔离
- **buildTool()** 的 fail-closed 工厂模式
- **三层梯度压缩** 的上下文管理策略
- **分区并发** 的安全且高效调度
- **34 行 Store** 的极简状态管理

这些决策不是孤立的——它们形成了一个连贯的系统，每个决策都在为其他决策创造条件。这就是架构设计的本质：**不是选择最好的技术，而是选择最能协同工作的技术组合**。
