# 07 - 多智能体与任务系统

## 1. 概述

Claude Code 不只是一个单智能体系统。它支持多种形式的智能体协作：
- **子代理**（AgentTool 生成，独立进程执行子任务）
- **协调器模式**（Coordinator Mode，领导者-工作者架构）
- **群组模式**（Swarm Mode，多智能体团队协作）
- **后台任务**（Dream、Cron、Workflow 等异步工作）

---

### 2.1 类型定义

```typescript
// src/Task.ts
type TaskType =

type Task = {

```

## 3. 任务实现（src/tasks/）

### 3.1 LocalShellTask

后台执行 shell 命令，不阻塞主 Agent Loop：
- 输出写入临时文件（`taskDiskOutput.ts`）
- 实时进度更新通过 `onProgress` 回调
- 支持长时间运行的命令（如 `npm test`）
- 可通过 `TaskStopTool` 终止

### 3.2 LocalAgentTask

子代理实现——fork 一个新的 Claude Code 进程：
- 使用 `forkSubagent.ts` 创建子进程
- 子进程有独立的上下文和工具集
- 结果通过文件系统或 IPC 回传
- 支持内存共享模式（`cache-sharing`）

### 3.3 InProcessTeammateTask

进程内队友——在同一个进程中运行的智能体：
- 共享主进程的 AppState
- 通过 inbox 消息队列通信
- 更轻量（无进程间通信开销）
- 适合紧密协作场景

### 3.4 DreamTask

后台思考任务——在空闲时让 LLM "思考"：
- 不消耗用户可见的 token 预算
- 可以预生成代码建议
- 在 `stopHooks.ts` 的 `executeAutoDream()` 中触发
- 结果作为 "灵感" 注入后续对话

### 3.5 RemoteAgentTask

通过 Bridge 运行的远程子代理：
- 在远程机器/容器上执行
- 通过 WebSocket 传输输入/输出
- 权限通过 `remotePermissionBridge.ts` 桥接

## 4. AgentTool（子代理生成器）

`src/tools/AgentTool/`（15 个文件）是最复杂的工具之一。

### 4.1 核心机制

```typescript
// 调用流程：
AgentTool.call({ prompt, agentType })
  │
  ├── 1. loadAgentsDir() → 加载可用 agent 定义
  ├── 2. resolveAgentType() → 解析 agent 类型
  ├── 3. forkSubagent() 或 spawnLocalProcess()
  │     ├── 构建子代理的系统提示词
  │     ├── 选择子代理的工具子集
  │     ├── 设置工作目录和环境变量
  │     └── 启动子进程 / 进程内 agent
  │
  ├── 4. 流式接收子代理输出
  │     ├── 文本 → yield 进度
  │     └── 工具调用 → yield 进度
  │
  └── 5. 收集最终结果
        ├── 成功 → 返回结果
        ├── 中断 → 返回部分结果
        └── 错误 → 返回错误信息
```

### 4.2 Agent 定义

`loadAgentsDir.ts` 从多个来源加载 agent 定义：

```
~/.claude/agents/              ← 全局自定义 agent
.claude/agents/                ← 项目级自定义 agent
内置 agent 定义                ← hardcoded in bundledAgents.ts
```

每个 agent 定义包含：
- 名称、描述、颜色
- 允许的工具列表
- 系统提示词
- 最大 turn 数

### 4.3 Agent 颜色管理

`agentColorManager.ts` 为每个智能体分配唯一颜色：
- 颜色池循环使用
- 颜色映射存储在 `bootstrap/state.ts` 的 `agentColorMap`
- UI 组件通过颜色区分不同智能体的输出

### 4.4 子代理上下文隔离

`createSubagentContext()` 创建与父代理隔离的上下文：
- 独立的 `readFileState`（文件状态缓存）
- 独立的 `fileHistoryState`（文件编辑历史）
- 独立的 `denialTracking`（权限拒绝追踪）
- **共享** `setAppStateForTasks`（任务注册/清理）

## 5. 协调器模式（Coordinator Mode）

`src/coordinator/coordinatorMode.ts`（369行）实现领导者-工作者架构：

### 5.1 架构

```
┌─────────────────────────────────────────────┐
│             Coordinator（领导者）               │
│                                             │
│  ┌─── System Prompt ───────────────────┐    │
│  │ 角色：任务分发和结果整合               │    │
│  │ 工具：Agent + TaskStop + SendMessage  │    │
│  │ 限制：不做具体编码工作                  │    │
│  └─────────────────────────────────────┘    │
│                                             │
│  调度策略：                                   │
│  1. 分析用户需求 → 分解为子任务               │
│  2. 为每个子任务选择合适的 worker              │
│  3. 通过 AgentTool 生成 worker               │
│  4. 等待结果 → 整合 → 继续下一步              │
└─────────────────────────────────────────────┘
        │ AgentTool.call()
        ▼
┌──────────────┐ ┌──────────────┐ ┌──────────────┐
│  Worker 1    │ │  Worker 2    │ │  Worker 3    │
│ (Bash/Read/  │ │ (Edit/Write) │ │ (Grep/Glob)  │
│  Execute)    │ │              │ │              │
└──────────────┘ └──────────────┘ └──────────────┘
```

### 5.2 系统提示词

协调器的系统提示词（369行）定义了严格的行为规范：
- **不做具体编码**：协调器只分配任务，不直接执行工具
- **任务分解**：每个子任务应该是独立的、可并行的
- **结果整合**：收集所有 worker 的结果，综合后回复用户
- **错误处理**：worker 失败时重新分配或回退

### 5.3 Worker 工具集

Worker 拥有受限的工具集：
- 不包含 AgentTool（防止 worker 再生成 worker）
- 不包含权限敏感工具（如 ConfigTool）
- 保留核心编码工具（Bash、Read、Edit、Write、Grep、Glob）

## 6. 群组模式（Swarm Mode）

`src/utils/swarm/`（14 个文件 + backends/）实现多智能体团队。

### 6.1 架构

```
┌─────────────────────────────────────────────────┐
│                  Team Lead                        │
│  ├── TeamCreateTool → 创建团队                     │
│  ├── SendMessageTool → 向成员发消息                 │
│  └── TeamDeleteTool → 解散团队                     │
│                                                   │
│  管理通过 src/utils/teammate.ts:                   │
│  ├── startTeammate() → 启动成员                    │
│  ├── stopTeammate() → 停止成员                     │
│  └── isTeammateAlive() → 心跳检测                  │
├──────────┬──────────┬──────────┬──────────────────┤
│ Member 1 │ Member 2 │ Member 3 │ ...              │
│ (独立进程) │ (进程内)  │ (Tmux)   │                  │
└──────────┴──────────┴──────────┴──────────────────┘
```

### 6.2 运行后端

Swarm 支持多种进程管理后端（`src/utils/swarm/backends/`）：

| 后端 | 适用场景 | 特点 |
|------|---------|------|
| `InProcessBackend` | 轻量协作 | 共享进程，低延迟，共享内存 |
| `TmuxBackend` | Mac/Linux 终端 | 独立 tmux 窗格，可见输出 |
| `iTerm2Backend` | Mac iTerm2 | 独立标签页，原生体验 |

### 6.3 通信机制

- **SendMessageTool**：向指定 agent 发送消息（通过 `inbox` 消息队列）
- **ListPeersTool**：列出当前可用的团队成员
- **MailboxProvider**（`context/mailbox.tsx`）：React 层面的消息投递
- **权限同步**：`permissionSync.ts` 确保团队成员的权限与领导者一致

## 7. 任务管理工具

### 7.1 TaskCreate/TaskGet/TaskUpdate/TaskList/TaskOutput/TaskStop

这是一套完整的任务 CRUD 工具（`src/tools/`）：

```
TaskCreateTool → 创建新任务（类型、描述、初始参数）
TaskGetTool    → 获取单个任务状态
TaskListTool   → 列出所有任务
TaskUpdateTool → 更新任务状态/描述
TaskOutputTool → 获取任务输出（支持流式）
TaskStopTool   → 终止任务（调用 task.kill()）
```

### 7.2 任务磁盘输出

`src/utils/task/taskDiskOutput.ts` 将大型任务输出写入临时文件：
- 避免内存中积累大量输出
- 支持流式读取（通过 `TaskOutputTool`）
- 任务完成后自动清理

## 8. 智能体间通信

### 8.1 Inbox 模型

```
AppState.inbox: Map<AgentId, UserMessage[]>

发送消息：
  SendMessageTool.call({ recipient, message })
    → setAppState(prev => ({
        ...prev,
        inbox: new Map([...prev.inbox, [recipientId, [...existing, newMessage]]])
      }))

接收消息：
  agent 的 pendingUserMessages 队列
    → 在 Agent Loop 每轮迭代开始时检查
    → 如果有新消息，注入为用户消息
```

### 8.2 权限桥接

`src/remote/remotePermissionBridge.ts` 处理远程智能体的权限请求：
- 远程智能体无法显示 UI 对话框
- 权限请求通过 WebSocket 发送到主进程
- 主进程显示对话框，用户决策后回传

## 9. 设计精髓

### 进程隔离 vs 进程内

Claude Code 支持两种智能体运行模式：
- **进程内**（`InProcessTeammateTask`）：共享 AppState，通过 inbox 通信，低延迟
- **进程外**（`LocalAgentTask`）：独立进程，文件系统通信，强隔离

这种双模式设计让用户可以根据场景选择：紧密协作用进程内（如多文件同步编辑），独立任务用进程外（如长时间运行的测试）。

### 智能体的工具子集

每个智能体获得受限的工具集——这是安全的关键。Worker 不能生成新的 Worker（防止无限递归），不能修改全局配置，不能访问敏感工具。工具集的选择是声明式的（在 agent 定义中），而不是命令式的（在运行时判断）。

### 后台任务的生命周期

任务从创建到完成/失败/终止，状态存储在 `AppState.tasks` 中。`stopTask.ts` 统一处理终止逻辑（发送 SIGTERM → 等待超时 → SIGKILL）。后台任务不会因为主进程的 Agent Loop 结束而被遗忘——它们的状态始终可在 UI 中查看。
