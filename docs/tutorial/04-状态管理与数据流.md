# 04 - 状态管理与数据流

## 1. 概述

Claude Code 的状态管理分为两个独立但协作的层：

1. **Bootstrap State**（`src/bootstrap/state.ts`）— 全局单例，存储会话级元数据（成本、Token 计数、Session ID 等），与 UI 无关
2. **AppState**（`src/state/`）— React 响应式状态，驱动整个终端 UI 渲染，使用自定义 Observable Store 模式

这种分离确保了核心业务逻辑（成本追踪、Token 计数）不依赖 React，而 UI 状态则享受 React 的高效渲染。

## 2. 极简 Store 模式（store.ts）

`src/state/store.ts`（仅 34 行）是整个状态管理的基石：

```typescript
// src/state/store.ts:10
export function createStore<T>(
  initialState: T,
  onChange?: OnChange<T>,
): Store<T> {
  let state = initialState
  const listeners = new Set<Listener>()

  return {
    getState: () => state,
    setState: (updater: (prev: T) => T) => {
      const prev = state
      const next = updater(prev)
      if (Object.is(next, prev)) return   // 引用相等性检查
      state = next
      onChange?.({ newState: next, oldState: prev })
      for (const listener of listeners) listener()
    },
    subscribe: (listener: Listener) => {
      listeners.add(listener)
      return () => listeners.delete(listener)
    },
  }
}
```

### 设计特点

| 特点 | 说明 |
|------|------|
| **不可变更新** | `setState` 接受 updater 函数，返回新对象（spread/替换） |
| **引用相等性** | `Object.is(next, prev)` 避免无变化时的无效通知 |
| **无中间件** | 无需 Redux middleware，状态更新是同步的 |
| **onChange 回调** | 状态变化时触发副作用（持久化、通知等） |
| **订阅退订** | `subscribe` 返回清理函数，兼容 React useEffect |

### 关键状态域

| 状态域 | 类型 | 用途 |
|--------|------|------|
| `toolPermissionContext` | `ToolPermissionContext` | 权限模式、允许/拒绝/询问规则 |
| `tasks` | `{ [taskId]: TaskState }` | 所有后台任务（shell、agent、dream） |
| `mcp` | MCP 状态 | MCP 服务器连接、工具、命令、资源 |

| `inbox` | `Map<AgentId, Message[]>` | 智能体间消息投递 |
| `speculation` | `SpeculationState` | 推测执行（提前执行可能的用户意图） |
| `todos` | `TodoList` | Todo 列表管理 |
| `replBridge*` | 多个字段 | Bridge 连接状态（URL、Session、错误等） |
| `remoteConnectionStatus` | 枚举 | 远程会话连接状态 |

### DeepImmutable 包装

`AppState` 被 `DeepImmutable<>` 包装（`AppStateStore.ts:89`），确保所有嵌套属性在类型层面不可变。唯一例外是 `tasks`（包含函数类型，无法深度冻结）和 `agentNameRegistry`（Map 的 mutation 由内部控制）。

## 4. React 集成（AppState.tsx）

`src/state/AppState.tsx` 将 Store 与 React 连接，核心是三个 Hook：

### 4.1 AppStateProvider

```typescript
// src/state/AppState.tsx:37
function AppStateProvider({ children, initialState, onChangeAppState }) {
  const [store] = useState(() => 
    createStore(initialState ?? getDefaultAppState(), onChangeAppState)
  )
  // ...效果：检查 bypass permissions、设置变更监听
  return (
    <HasAppStateContext.Provider value={true}>
      <AppStoreContext.Provider value={store}>
        <MailboxProvider>
          <VoiceProvider>{children}</VoiceProvider>
        </MailboxProvider>
      </AppStoreContext.Provider>
    </HasAppStateContext.Provider>
  )
}
```

Provider 层级：`AppStateProvider` > `AppStoreContext` > `MailboxProvider` > `VoiceProvider`

### 4.2 useAppState(selector) — 选择性订阅

```typescript
// src/state/AppState.tsx:142
export function useAppState(selector) {
  const store = useAppStore()
  const get = () => selector(store.getState())
  return useSyncExternalStore(store.subscribe, get, get)
}
```

使用 `useSyncExternalStore` 实现 **Selector 级别的细粒度订阅**。只有 selector 返回值变化时（`Object.is` 比较），组件才会重渲染。

**用法示例**：

```typescript
// 只在 verbose 变化时重渲染
const verbose = useAppState(s => s.verbose)

// 只在权限模式变化时重渲染
const mode = useAppState(s => s.toolPermissionContext.mode)

// 多个独立字段 = 多次调用
const verbose = useAppState(s => s.verbose)
const model = useAppState(s => s.mainLoopModel)
```

**反模式**：不要返回新对象（`Object.is` 会认为变化了）：

```typescript
// ❌ 错误：每次调用都创建新对象
const { text, promptId } = useAppState(s => ({ text: s.xxx, promptId: s.yyy }))

// ✅ 正确：返回已有引用
const { text, promptId } = useAppState(s => s.promptSuggestion)
```

### 4.3 useSetAppState() — 无订阅更新

```typescript
// src/state/AppState.tsx:170
export function useSetAppState() {
  return useAppStore().setState
}
```

返回稳定的 `setState` 引用——使用此 Hook 的组件 **永远不会因状态变化而重渲染**。适用于只需要更新状态、不需要读取状态的场景（如事件处理函数）。

## 5. 状态变更副作用（onChangeAppState.ts）

`onChangeAppState`（`src/state/onChangeAppState.ts`）是 Store 创建时传入的回调，作为状态变更的 **统一观察点**：

```
setAppState(updater)
    │
    ▼
store.setState(updater)
    │
    ├── Object.is(next, prev) === true → 无变化，跳过
    │
    ├── 通知所有 listeners → React 重渲染
    │
    └── onChangeAppState({ newState, oldState })
         │
         ├── permissionMode 变化 → notifySessionMetadataChanged(CCR)
         │                    → notifyPermissionModeChanged(SDK)
         │
         ├── mainLoopModel 变化 → updateSettingsForSource
         │                    → setMainLoopModelOverride
         │
         ├── expandedView 变化 → saveGlobalConfig
         │
         ├── verbose 变化 → saveGlobalConfig
         │
         └── settings 变化 → clearApiKeyHelperCache
                          → clearAwsCredentialsCache
                          → clearGcpCredentialsCache
                          → applyConfigEnvironmentVariables
```

这种 **单点观察** 设计（`onChangeAppState.ts:65-92`）解决了之前模式变更不同步的问题——任何 `setAppState` 路径改变权限模式，都会自动通知 CCR 和 SDK。

### 为什么分两层

1. **Bootstrap State 不需要 React**——成本追踪、Token 计数在 `query()` 的 AsyncGenerator 中更新，此时可能没有 React 上下文
2. **AppState 需要 React**——权限对话框、任务面板、模型选择器等 UI 组件需要响应式渲染
3. **避免循环依赖**——Bootstrap 是 import DAG 的叶子节点，AppState 可以依赖 Bootstrap 但反之不行

## 7. 数据流全景图

```
用户操作（输入、点击、快捷键）
    │
    ▼
React Event Handler
    │
    ├── useSetAppState()(prev => ({...prev, field: newValue}))
    │       │
    │       ▼
    │   store.setState(updater)
    │       │
    │       ├── Object.is 检查 → 无变化则跳过
    │       │
    │       ├── 更新 state 引用
    │       │
    │       ├── onChangeAppState({ newState, oldState })
    │       │     ├── 权限模式变化 → CCR/SDK 通知
    │       │     ├── 模型变化 → settings 持久化
    │       │     └── 设置变化 → 缓存清理
    │       │
    │       └── 通知所有 listeners
    │             │
    │             ▼
    │         React useSyncExternalStore
    │             │
    │             ├── selector(oldState) !== selector(newState)?
    │             │     ├── 是 → 组件重渲染
    │             │     └── 否 → 跳过
    │             │
    │             ▼
    │         Ink 渲染引擎 → 终端输出更新
    │
    └── Bootstrap State 直接更新（无重渲染）
          ├── addToTotalCostState(cost, usage, model)
          ├── addToTotalDurationState(duration, durationWithoutRetries)
          └── setCwdState(cwd)
```

## 9. 设计精髓

### 为什么不用 Redux/Zustand

Claude Code 的 Store 实现仅 34 行，却满足了所有需求：

- **不需要 DevTools**：终端 UI 不需要时间旅行调试
- **不需要中间件**：所有状态更新是同步的，`onChange` 回调替代了 middleware
- **不需要序列化**：终端 UI 不需要状态快照/恢复
- **更好的类型推导**：直接 `store.getState()` 比 `useSelector(state => ...)` 更精确
- **零依赖**：不引入任何第三方状态管理库

### Selector 模式的性能优化

`useAppState(selector)` 使用 `useSyncExternalStore` 实现 **选择器级别的细粒度订阅**。在 144 个 UI 组件、85 个 Hook 的大型应用中，这意味着：

- 只有真正关心某个字段的组件才会在该字段变化时重渲染
- `useSetAppState()` 返回的 updater 永远不会触发重渲染
- `Object.is` 引用相等性检查避免了深比较的开销

### 单点副作用观察

`onChangeAppState` 作为唯一的副作用观察点，确保了：
- **一致性**：无论状态从哪个路径更新，副作用都会执行
- **可审计**：所有持久化和通知逻辑集中在一个文件
- **防遗漏**：之前散布在 8+ 个调用点的权限模式通知，现在只需在一个 diff 检查
